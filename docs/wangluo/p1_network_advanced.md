# 一、局域网 vs 广域网 vs 互联网

你可能听过这几个名词，它们之间是什么关系？

| 网络类型 | 英文 | 范围 | 例子 |
|---------|------|------|------|
| **局域网** | LAN (Local Area Network) | 小范围（一栋楼、一个办公室） | 你家的 WiFi 网络、机房内网 |
| **广域网** | WAN (Wide Area Network) | 大范围（跨城市、跨国家） | 电信/联通/移动的骨干网 |
| **互联网** | Internet | 全球 | 由无数 LAN 和 WAN 互联而成 |

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  你家 WiFi   │    │  公司内网    │    │  学校网络    │
│  (局域网)    │    │  (局域网)    │    │  (局域网)    │
└──────┬──────┘    └──────┬──────┘    └──────┬──────┘
       │                  │                  │
       └──────── 广域网（运营商骨干网）────────┘
                         │
                    ┌────┴────┐
                    │ 互联网   │
                    │(Internet)│
                    └─────────┘
```

> **类比**：
> - 局域网 = 你家内部的对讲系统
> - 广域网 = 城市间的电话网络
> - 互联网 = 全世界的电话网络互联

# 二、NAT — 公网 IP 不够用怎么办？

## 2.1 问题：IPv4 地址枯竭

IPv4 地址只有约 43 亿个（$2^{32}$），而全球有数十亿台设备。如果每台设备都需要一个公网 IP，根本不够用。

## 2.2 解决方案：NAT

**NAT（Network Address Translation，网络地址转换）** 让多个设备**共享一个公网 IP**。

你家路由器就在做 NAT：

```
你的手机  192.168.1.101  ─┐
你的电脑  192.168.1.102  ─┤   路由器    公网IP: 203.0.113.5  ──► 互联网
你的平板  192.168.1.103  ─┤  (NAT)
智能电视  192.168.1.104  ─┘
```

当你的手机访问百度时：
1. 手机发出数据包：`192.168.1.101 → 百度服务器`
2. 路由器把源地址替换为公网 IP：`203.0.113.5 → 百度服务器`
3. 百度回复到 `203.0.113.5`
4. 路由器根据记录，把回复转发给 `192.168.1.101`（你的手机）

> **类比：公司前台**
> 
> 外部快递寄到公司总机（公网 IP），前台（路由器）根据收件人姓名（端口号 + 记录表）把快递分发给具体的员工（内网设备）。

## 2.3 NAT 的影响

NAT 解决了 IP 不够用的问题，但也带来一个限制：**外部无法主动访问内网设备**。

这就是为什么你在家用 Flask 启动了服务器，外网同学无法直接访问——因为路由器的 NAT 挡住了外部请求。解决方法包括：
- **端口转发**：在路由器上配置，把特定端口的请求转发给内网设备。
- **内网穿透工具**：如 ngrok、frp、花生壳等。

::: info 动手试试
1. 打开百度搜索"IP"，会显示你的**公网 IP**
2. 用 `ipconfig` 查看你的**内网 IP**
3. 两者不同——这就是 NAT 在工作的证据
:::

# 三、子网掩码与 CIDR

## 3.1 子网掩码是什么？

你用 `ipconfig` 时经常看到 `255.255.255.0`，这就是**子网掩码**。它用来告诉设备：IP 地址中哪部分是"网络号"，哪部分是"主机号"。

```
IP 地址：    192.168.1  .105
子网掩码：   255.255.255.0
                      ↑
            网络部分   │主机部分
```

当子网掩码是 `255.255.255.0` 时：
- 前三段 `192.168.1` 是**网络号**（同一子网的所有设备都一样）
- 最后一段 `.105` 是**主机号**（同一子网内每个设备不同）
- 这个子网最多有 $256 - 2 = 254$ 台主机（去掉全 0 和全 1）

## 3.2 CIDR 表示法

你可能见过 `192.168.1.0/24` 这样的写法，这叫做 **CIDR（无类别域间路由）** 表示法。

`/24` 表示子网掩码的前 24 位是 1（即 `255.255.255.0`）。

| CIDR | 子网掩码 | 可用主机数 | 常见用途 |
|------|---------|-----------|---------|
| `/24` | 255.255.255.0 | 254 | 家庭/小型办公网络 |
| `/16` | 255.255.0.0 | 65,534 | 大型企业网络 |
| `/8` | 255.0.0.0 | 16,777,214 | 超大网络 |

::: tip 简单理解
`/24` = 最后一段（8 位）可以变化 = 最多 254 台设备
`/16` = 最后两段（16 位）可以变化 = 最多 65534 台设备
数字越小，子网越大。
:::

# 四、防火墙 — 网络世界的"门卫"

## 4.1 什么是防火墙？

**防火墙（Firewall）** 是网络安全的"门卫"，它根据预设的规则决定哪些网络流量可以通行，哪些要被拦截。

> 类比：小区保安——只允许业主和登记访客进入，其他人一律不让进。

## 4.2 防火墙工作在哪里？

防火墙可以在多个位置：
- **路由器上**：你家路由器就内置了简单的防火墙功能
- **操作系统上**：Windows 防火墙、Linux 的 iptables
- **服务器上**：云服务商的安全组（如阿里云安全组）

## 4.3 常见防火墙规则

| 规则 | 效果 |
|------|------|
| 只开放 80 和 443 端口 | 只允许 HTTP/HTTPS 流量 |
| 禁止外部访问 27017 端口 | MongoDB 只允许本机访问 |
| 禁止特定 IP 段 | 屏蔽某个地区的访问 |

::: info 为什么你的 Flask 服务别人访问不了？
如果别人说访问不了你的 Flask 服务，除了 NAT 之外，还可能是**防火墙**在拦截。Windows 防火墙默认会阻止外部访问非标准端口（如 5000）。
:::

# 五、代理与 VPN

## 5.1 代理（Proxy）

**代理** 就是一个"中间人"——你不直接访问目标服务器，而是让代理服务器帮你访问。

```
你的电脑 ──► 代理服务器 ──► 目标服务器
```

> 类比：你不能出国买东西，就让代购帮你买。

常见用途：
- 访问被限制的内容
- 隐藏真实 IP
- 加速访问（缓存）

## 5.2 VPN（虚拟专用网络）

**VPN** 在你的设备和 VPN 服务器之间建立一条**加密隧道**，所有流量都通过这条隧道传输。

```
你的电脑 ═══加密隧道═══ VPN服务器 ──► 目标服务器
```

> 类比：在公共道路上建了一条只有你能走的密封管道。

VPN 和代理的区别：
- **代理**：通常只代理特定应用（如浏览器）的流量
- **VPN**：代理整台设备的所有网络流量，并且全部加密

# 六、CDN — 让内容离用户更近

## 6.1 什么是 CDN？

**CDN（Content Delivery Network，内容分发网络）** 是一组分布在全球各地的服务器，它们缓存了网站的静态内容（图片、CSS、JS 文件等），让用户从**距离最近**的服务器获取资源。

> **类比：连锁超市 vs 中央仓库**
> 
> 如果全国只有一个中央仓库（源服务器），所有人买东西都要从北京发货，偏远地区可能要等好几天。CDN 就像在全国各地开了连锁超市（边缘节点），你从最近的超市拿货，速度自然快。

```
没有 CDN：
用户(长沙) ──────────────► 源服务器(北京)    延迟 50ms

有 CDN：
用户(长沙) ──► CDN节点(长沙)    延迟 5ms
                 ↕ 自动同步
              源服务器(北京)
```

## 6.2 CDN 的工作原理

1. 网站把静态资源交给 CDN 服务商。
2. CDN 把资源缓存到全球各地的节点。
3. 用户请求资源时，DNS 把用户导向最近的节点。
4. 用户从最近的节点获取资源，速度飞快。

::: tip 你已经在用 CDN 了
当你 `nslookup www.baidu.com` 时，返回的 IP 地址可能每次都不同——这就是 CDN 在根据你的地理位置返回最近的服务器。
:::

# 七、Cookie、Session、Token — 如何记住"你是谁"？

## 7.1 问题：HTTP 是"健忘"的

HTTP 协议本身是**无状态**的——服务器不记得你是谁。每次请求都像是"第一次见面"。

那登录状态是怎么保持的？答案是**在请求中携带身份凭证**。

## 7.2 Cookie

**Cookie** 是服务器通过响应头 `Set-Cookie` 存在你浏览器里的一小段数据。之后每次请求，浏览器都会自动带上这些 Cookie。

```
第1次请求：
浏览器 → 服务器：我要登录
浏览器 ← 服务器：Set-Cookie: session_id=abc123

第2次请求：
浏览器 → 服务器：我要看个人主页（Cookie: session_id=abc123）
浏览器 ← 服务器：这是你的个人主页，张三
```

> 类比：你去图书馆办了张借书证（Cookie），以后每次去只要出示借书证，管理员就知道你是谁。

## 7.3 Session

**Session** 是保存在服务器端的用户数据。浏览器通过 Cookie 中的 `session_id` 来告诉服务器"我是谁"。

```
浏览器（Cookie: session_id=abc123）
         │
         ▼
服务器查找 session 存储：
   abc123 → { username: "张三", role: "admin" }
         │
         ▼
返回张三的数据
```

## 7.4 Token（JWT）

**Token** 是一种不需要服务器存储的认证方式。服务器把用户信息加密成一个字符串（Token）发给客户端，客户端每次请求时带上它。

```
登录成功后：
服务器 → 浏览器：eyJhbGciOiJIUzI1NiIs...（一个很长的字符串）

后续请求：
浏览器 → 服务器：Authorization: Bearer eyJhbGciOiJIUzI1NiIs...
```

> 类比：Cookie/Session 像借书证（图书馆需要登记），Token 像身份证（自带信息，不需要对方登记）。

| 方式 | 存储位置 | 服务端是否需要存储 | 适用场景 |
|------|---------|------------------|---------|
| **Cookie** | 浏览器 | 不需要（数据在 Cookie 里） | 简单状态 |
| **Session** | 服务器 | ✅ 需要 | 传统 Web 应用 |
| **Token (JWT)** | 客户端 | ❌ 不需要 | API、前后端分离、移动端 |

# 八、常见网络攻击简介

了解一下常见的网络攻击，不是为了学黑客，而是为了**知道风险在哪里**，写出更安全的代码。

## 8.1 DNS 劫持

攻击者篡改 DNS 解析结果，把你本来要访问的网站导向恶意服务器。

```
正常：www.bank.com → 1.2.3.4（真正的银行网站）
被劫持：www.bank.com → 5.6.7.8（钓鱼网站，长得一模一样）
```

**防范**：使用 HTTPS（即使 DNS 被劫持，证书验证也会报警）。

## 8.2 中间人攻击（MITM）

攻击者在你和服务器之间"偷听"甚至"篡改"通信内容。

```
正常：你 ←──────────────→ 服务器
被攻击：你 ←──→ 攻击者 ←──→ 服务器
```

**防范**：使用 HTTPS + 不连接不可信的公共 WiFi。

## 8.3 DDoS 攻击

**DDoS（分布式拒绝服务攻击）** 就是用大量假流量"淹没"目标服务器，让它无法为正常用户提供服务。

> 类比：恶意组织派了 10000 个人同时拨打一家餐厅的电话订座，真正的顾客就打不进去了。

**防范**：使用 CDN、云防火墙、流量清洗服务。

## 8.4 SQL 注入（与后端开发直接相关）

如果后端代码直接把用户输入拼接到 SQL 查询中，攻击者可以通过精心构造的输入来执行恶意 SQL。

```python
# ❌ 危险写法（字符串拼接）
query = f"SELECT * FROM users WHERE name = '{user_input}'"

# 如果 user_input 是: ' OR '1'='1
# 实际执行的 SQL 变成了：
# SELECT * FROM users WHERE name = '' OR '1'='1'
# 这会返回所有用户数据！
```

```python
# ✅ 安全写法（参数化查询）
cursor.execute("SELECT * FROM users WHERE name = %s", (user_input,))
```

**防范**：**永远不要拼接用户输入到 SQL 中**，使用参数化查询或 ORM。

::: warning 安全提醒
作为后端开发者，你写的每一行代码都可能成为攻击入口。记住三条原则：
1. **永远不信任用户输入** — 所有来自前端的数据都要验证和清洗
2. **永远不硬编码密码/密钥** — 使用环境变量
3. **永远使用 HTTPS** — 至少在生产环境中
:::

# 九、小结

本篇介绍了真实网络环境中的重要概念：

| 概念 | 一句话总结 |
|------|-----------|
| **LAN/WAN/Internet** | 局域网 → 广域网 → 互联网，层层互联 |
| **NAT** | 多台设备共享一个公网 IP，路由器负责"翻译" |
| **子网掩码/CIDR** | 划分网络中哪些设备属于同一"小区" |
| **防火墙** | 根据规则允许或拒绝网络流量 |
| **代理/VPN** | 通过中间人访问目标，VPN 还提供加密 |
| **CDN** | 全球分布的缓存节点，让用户就近获取资源 |
| **Cookie/Session/Token** | HTTP 无状态的解决方案，记住"你是谁" |
| **网络攻击** | DNS 劫持、中间人、DDoS、SQL 注入——知道风险才能防范 |
