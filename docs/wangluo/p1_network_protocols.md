# 网络协议与分层模型：数据是怎么"打包"传输的？

上一篇我们了解了 IP 地址、域名、端口号等基础概念。但你有没有想过：**数据在网络中到底是怎么传输的？** 一封邮件、一个网页、一段视频，它们在网络中"长什么样"？

本篇将带你理解网络的"游戏规则"——协议，以及数据传输的"分层打包"思想。

::: tip 前置知识
建议先阅读 [网络世界的第一步](./p1_network_basics)，了解 IP 地址、域名、端口等基本概念。
:::

# 一、为什么需要协议？

想象一下，如果全世界的快递公司没有统一的快递单格式——顺丰用 A4 纸、京东用卡片、中通用短信……那快递系统就会彻底混乱。

网络世界也一样。全世界有数十亿台设备，来自不同厂商、运行不同系统，它们之所以能互相通信，就是因为大家遵守了**同一套规则**——**协议（Protocol）**。

> **协议就是通信双方约定好的"语言和格式"**。
> 
> 类比：就像中国人之间说中文，国际上用英文。网络设备之间用的是 TCP/IP、HTTP 等协议。

# 二、为什么要分层？

## 2.1 一个生活类比：寄信

当你给朋友寄一封信时，实际发生了这些事：

```
第4层（你）    ：写好信的内容（"周六见！"）
第3层（封信封） ：写上收件人地址、寄件人地址
第2层（贴邮票） ：贴邮票、投进邮筒
第1层（运输）   ：邮车把信送到目的地城市
```

每一层只关心自己的事情：
- 你只管写信，不需要知道邮车走哪条路。
- 邮车只管运输，不需要知道信里写了什么。

网络的分层模型就是这个思想：**每一层负责一件事，层与层之间通过标准接口传递数据**。

## 2.2 分层的好处

| 好处 | 说明 |
|------|------|
| **各司其职** | 每层只关注自己的任务，降低复杂度 |
| **灵活替换** | 换一条网线（物理层），不影响上面的应用 |
| **易于维护** | 出问题时可以逐层排查 |
| **标准化** | 不同厂商可以各自实现某一层，只要接口一致就能协作 |

# 三、TCP/IP 四层模型详解

互联网使用的分层模型叫做 **TCP/IP 模型**，分为四层。数据从上往下传输时，每一层会加上自己的"包装"（头部信息），这个过程叫做 **封装（Encapsulation）**。

```
┌─────────────────────────────────────────────────────────────┐
│                         应用层                               │  ← HTTP、DNS、FTP、SSH
│                  (你看到的数据：网页、邮件)                     │
├─────────────────────────────────────────────────────────────┤
│                         传输层                               │  ← TCP、UDP
│                   (保证数据可靠到达)                          │
├─────────────────────────────────────────────────────────────┤
│                         网络层                               │  ← IP
│                  (找到目标设备在哪)                           │
├─────────────────────────────────────────────────────────────┤
│                         链路层                               │  ← 以太网、WiFi
│                  (在物理介质上传输比特)                       │
└─────────────────────────────────────────────────────────────┘
```

数据封装过程（类比寄信）：

```
你的消息               "周六见！"
    ↓ 封信封（加上地址）     ←  传输层加 TCP 头
    ↓ 贴邮票（加路由信息）   ←  网络层加 IP 头
    ↓ 放进快递盒            ←  链路层加帧头帧尾
    ↓ 通过快递车发出         ←  物理信号传输
```

## 3.1 链路层 — "最后一公里"

**负责**：在同一局域网内的设备之间传输数据。

> 类比：小区内的快递员，负责把包裹从小区门口送到你家门口。

- 使用 **MAC 地址** 来识别同一网络中的设备。
- 数据以 **帧（Frame）** 为单位传输。
- 常见技术：以太网（有线）、WiFi（无线）。

## 3.2 网络层 — "选择路线"

**负责**：把数据从源设备送到目标设备，可能跨越多个网络。

> 类比：快递分拣中心，决定包裹从北京到长沙走哪条路线。

- 使用 **IP 地址** 来标识设备。
- 数据以 **包（Packet）** 为单位传输。
- 核心设备是 **路由器**，它根据 IP 地址决定数据该往哪个方向转发。

```
你的电脑 ──► 路由器 ──► 运营商路由器 ──► ... ──► 目标服务器
            (转发)      (转发)                  (到达)
```

## 3.3 传输层 — "保证送达"

**负责**：确保数据完整、有序地从一个应用程序传到另一个应用程序。

> 类比：快递的"签收确认"服务——不仅要送到，还要确认对方收到了。

传输层有两个主要协议：**TCP** 和 **UDP**。

### TCP — 可靠传输

TCP（Transmission Control Protocol，传输控制协议）的特点：
- **可靠**：保证数据不丢失、不重复、按顺序到达。
- **有连接**：通信前需要先"握手"建立连接，通信后要"挥手"断开。
- **慢但稳**：因为要做各种确认和重传。

**TCP 三次握手**（建立连接）：

```
客户端                         服务器
  │                              │
  │  ── 1. SYN: "你好，在吗？" ──► │
  │                              │
  │ ◄── 2. SYN+ACK: "在的，你呢？"─│
  │                              │
  │  ── 3. ACK: "我也在，开始吧！"─►│
  │                              │
  │  ═══ 连接建立，开始传数据 ═══  │
```

> 类比：打电话
> 1. 你拨号（SYN）
> 2. 对方接听说"喂"（SYN+ACK）
> 3. 你说"你好，我是小明"（ACK）
> 4. 通话开始

**TCP 四次挥手**（断开连接）：

```
客户端                         服务器
  │                              │
  │  ── 1. FIN: "我说完了" ──────►│
  │                              │
  │ ◄── 2. ACK: "知道了" ────────│
  │                              │
  │ ◄── 3. FIN: "我也说完了" ────│
  │                              │
  │  ── 4. ACK: "好的，拜拜" ────►│
  │                              │
  │  ═══ 连接断开 ═══            │
```

> 为什么是四次而不是三次？因为双方可能不是同时说完的，需要各自确认"我说完了"。

### UDP — 快速传输

UDP（User Datagram Protocol，用户数据报协议）的特点：
- **不可信**：不保证数据一定能到达，也不保证顺序。
- **无连接**：不需要握手，直接发。
- **快但可能丢**：省去了确认和重传的开销。

### TCP vs UDP 对比

| 特性 | TCP | UDP |
|------|-----|-----|
| **可靠性** | ✅ 可靠，保证送达 | ❌ 可能丢包 |
| **速度** | 较慢（有确认机制） | 较快（无额外开销） |
| **连接方式** | 面向连接（三次握手） | 无连接 |
| **数据顺序** | ✅ 保证顺序 | ❌ 可能乱序 |
| **适用场景** | 网页、邮件、文件传输 | 视频通话、游戏、直播 |
| **类比** | 挂号信（有回执） | 明信片（投出去就不管了） |

::: tip 什么时候用哪个？
- 需要数据**完整准确**到达 → 用 TCP（如网页浏览、发邮件、下载文件）
- 需要**速度快**，偶尔丢几个包无所谓 → 用 UDP（如视频通话、在线游戏、直播）
:::

## 3.4 应用层 — "你看到的一切"

**负责**：直接为应用程序提供服务。

> 类比：快递送到后，你拆开包裹看到的东西。

我们日常接触的大部分协议都在这一层：

| 协议 | 全称 | 用途 | 类比 |
|------|------|------|------|
| **HTTP** | HyperText Transfer Protocol | 网页浏览 | 看报纸 |
| **HTTPS** | HTTP Secure | 加密网页浏览 | 看加密文件 |
| **DNS** | Domain Name System | 域名解析 | 查电话簿 |
| **FTP** | File Transfer Protocol | 文件传输 | 寄大包裹 |
| **SSH** | Secure Shell | 远程登录服务器 | 远程开门 |
| **SMTP** | Simple Mail Transfer Protocol | 发送邮件 | 寄信 |
| **WebSocket** | — | 实时双向通信 | 打电话 |

# 四、数据的完整旅程

让我们把分层模型串起来，看一个完整的数据传输过程：

```
你的电脑                                              百度服务器
┌──────────┐                                      ┌──────────┐
│  应用层   │ HTTP 请求: GET /index.html            │  应用层   │
├──────────┤                                      ├──────────┤
│  传输层   │ TCP 头: 源端口 54321 → 目标端口 80    │  传输层   │
├──────────┤                                      ├──────────┤
│  网络层   │ IP 头: 192.168.1.5 → 220.181.38.148  │  网络层   │
├──────────┤                                      ├──────────┤
│  链路层   │ 帧头: MAC 地址信息                    │  链路层   │
└──────────┘                                      └──────────┘
     │                    互联网                      │
     └──────────────────────────────────────────────┘
```

数据每经过一层就被"加一层包装"（封装），到达目的地后再逐层"拆包装"（解封装），最终还原成原始数据。

# 五、HTTPS 到底比 HTTP 安全在哪？

## 5.1 HTTP 的问题

普通 HTTP 传输的数据是**明文**的，就像寄一张明信片——邮递员（中间人）可以看到上面的所有内容。

在公共 WiFi 下，黑客甚至可以直接"偷看"你和服务器之间的所有通信。

## 5.2 HTTPS 的解决方案

HTTPS = HTTP + **TLS（传输层安全协议）**，它做了三件事：

| 功能 | 说明 | 类比 |
|------|------|------|
| **加密** | 数据被加密传输，第三方看不懂 | 把明信片装进信封 |
| **身份验证** | 通过数字证书确认服务器的身份 | 确认快递员有工牌 |
| **完整性** | 确保数据在传输过程中没有被篡改 | 信封有封口蜡，被拆过会留下痕迹 |

## 5.3 TLS 简化握手过程

```
浏览器                                    服务器
  │                                          │
  │ 1. "你好，我支持这些加密方式"              │
  │ ────────────────────────────────────────► │
  │                                          │
  │ 2. "好的，用这种方式，这是我的证书"        │
  │ ◄────────────────────────────────────────│
  │                                          │
  │ 3. [验证证书有效性]                       │
  │                                          │
  │ 4. "OK，这是加密密钥"                     │
  │ ────────────────────────────────────────► │
  │                                          │
  │ 5. ═══ 加密通道建立，开始安全通信 ═══      │
```

::: info 怎么判断网站是否使用 HTTPS？
看浏览器地址栏：
- 🔒 锁头图标 → HTTPS（安全）
- ⚠️ "不安全" 或没有锁头 → HTTP（不安全）
:::

# 六、WebSocket — 实时通信

## 6.1 HTTP 的局限

HTTP 是"请求-响应"模式：**客户端问一次，服务器答一次**。就像发短信——你发一条，等对方回一条。

但有些场景需要"持续对话"：
- 聊天应用（微信网页版）
- 实时游戏
- 股票行情推送
- 协作编辑（多人同时编辑文档）

如果用 HTTP，客户端就必须不停地"轮询"（每隔几秒问一次服务器："有新消息吗？"），效率很低。

## 6.2 WebSocket 的解决方案

WebSocket 在客户端和服务器之间建立一条**持久的双向通道**，双方可以随时互发消息。

> **类比：HTTP 是发短信，WebSocket 是打电话**
> 
> 短信（HTTP）：你发一条，等回复，再发一条……
> 电话（WebSocket）：接通后，双方随时可以说话。

```
HTTP 轮询：
客户端: 有新消息吗？ → 服务器: 没有
客户端: 有新消息吗？ → 服务器: 没有
客户端: 有新消息吗？ → 服务器: 有一条！     ← 大量无效请求

WebSocket：
客户端: 建立连接 → 服务器: 连接成功
    ... 双方保持连接 ...
服务器: 推送！新消息来了     ← 服务器主动推送，无需轮询
服务器: 推送！又一条新消息
```

| 对比 | HTTP 轮询 | WebSocket |
|------|----------|-----------|
| **通信方式** | 客户端定期询问 | 服务器主动推送 |
| **连接** | 每次请求都建立新连接 | 一次连接，持续使用 |
| **实时性** | 差（取决于轮询间隔） | 好（毫秒级延迟） |
| **资源消耗** | 高（大量无效请求） | 低（只需维持一个连接） |

# 七、常见协议速查表

| 协议 | 端口 | 传输层 | 用途 | 一句话说明 |
|------|------|--------|------|-----------|
| HTTP | 80 | TCP | 网页浏览 | 最基础的网页协议 |
| HTTPS | 443 | TCP | 安全网页浏览 | HTTP + 加密 |
| DNS | 53 | UDP/TCP | 域名解析 | 把域名翻译成 IP |
| FTP | 20/21 | TCP | 文件传输 | 传文件用的 |
| SSH | 22 | TCP | 远程登录 | 安全地控制远程服务器 |
| SMTP | 25 | TCP | 发邮件 | 寄信的协议 |
| POP3 | 110 | TCP | 收邮件 | 取信的协议 |
| IMAP | 143 | TCP | 收邮件 | 取信（更现代，支持同步） |
| WebSocket | 同HTTP | TCP | 实时通信 | 双向持久连接 |
| DHCP | 67/68 | UDP | 自动分配 IP | 设备连上网络时自动获取 IP |

# 八、小结

```
┌─────────────────────────────────────────────────────┐
│                     应用层                           │
│   HTTP / HTTPS / DNS / FTP / SSH / WebSocket        │
│   → 你看到的网页、邮件、文件                          │
├─────────────────────────────────────────────────────┤
│                     传输层                           │
│   TCP（可靠） / UDP（快速）                           │
│   → 保证数据到达对的应用程序                          │
├─────────────────────────────────────────────────────┤
│                     网络层                           │
│   IP                                                │
│   → 通过 IP 地址找到目标设备                          │
├─────────────────────────────────────────────────────┤
│                     链路层                           │
│   以太网 / WiFi                                     │
│   → 在物理介质上传输数据                              │
└─────────────────────────────────────────────────────┘
```

记住这个口诀：**应用层写内容，传输层保送达，网络层找路线，链路层跑腿**。
